# GPR68 interpolated-path ("string-method-adjacent") results

Generated by [`examples/gpcr_pipeline_gpr68_string_demo.py`](../../gpcr_pipeline_gpr68_string_demo.py).
This is the requested alternative to `gpcr_pipeline_gpr68_demo.py`'s
ANM-around-each-endpoint approach: instead of sampling *near* the active
and inactive basins separately, it builds a path *between* them, to fill
the gap that run's README flagged as the main remaining limitation.

## Method, and exactly what this is/isn't

The real GPR68 active and inactive models have **exact atom
correspondence** (365/365 residues, identical atom names per residue,
verified at the top of the script) and are already in the same reference
frame (same GPCRdb AFMS modeling pipeline/template) -- both preconditions
required for direct linear Cartesian interpolation to be meaningful (no
structural superposition step, which would remove the genuine rigid-body
helix motion that *is* the activation transition).

11 images were built at evenly spaced interpolation fractions f=0.0 (pure
active), 0.1, 0.2, ..., 1.0 (pure inactive). Each image was passed through
tool 3's real minimizer (150 L-BFGS iterations, same setting as every
other run this session) and scored with the same real AMBER ff14SB + GBn2
+ RRHO Gibbs free energy, at pH 7.4.

**This is linear interpolation with light local relaxation, not a
converged string method.** A true string method iteratively relaxes the
path with the path-tangent force component removed (so images can't just
roll downhill into one of the two endpoint basins) and periodically
reparametrizes images to stay evenly spaced along the *actual* minimum
free energy path -- multiple relaxation + reparametrization rounds, far
more force evaluations than this single-pass approach. What's implemented
here is the standard starting guess for a real string method ("linear
interpolation in Cartesians", LIC), scored once with a real force field --
a reasonable, honestly-labeled middle ground given the compute budget.

## Results

### The path did NOT collapse into either basin -- verified, not assumed

`rmsd_sanity_check.png`: RMSD-to-active and RMSD-to-inactive cross exactly
at f=0.5 (4.71 A each, half of the 9.42 A endpoint separation) and vary
essentially linearly with f. This confirms the 150-iteration local
relaxation only relieved steric strain from the naive interpolation --
it did not pull images back toward either endpoint's basin, which is
the standard failure mode this whole approach risks without proper
tangent-force removal. That the path held up anyway is a genuinely useful
result, not something to take for granted.

### A real, clean barrier -- `gibbs_vs_path.png`

| f | G (kcal/mol) |
|---|---|
| 0.0 (active) | -13897.0 |
| 0.1 | -13915.1 |
| 0.2 | -13921.6 (local minimum) |
| 0.3 | -13902.6 |
| 0.4 | -13857.5 |
| **0.5** | **-13784.8 (peak / candidate barrier)** |
| 0.6 | -13827.2 |
| 0.7 | -13879.0 |
| 0.8 | -13903.6 |
| 0.9 | -13890.9 |
| 1.0 (inactive) | -13916.5 |

G rises smoothly and roughly symmetrically to a peak of **-13784.8
kcal/mol exactly at the midpoint** (f=0.5), **137 kcal/mol** above the
lowest point on the path (f=0.2). Both endpoints (f=0.0, f=1.0) land close
to each other (-13897.0 vs. -13916.5) and close to what
`gpcr_pipeline_gpr68_demo.py`'s independent runs found for the
active/inactive basins directly, which is a useful cross-check that these
numbers are internally consistent across the two different demos. The
profile isn't perfectly smooth (small local structure around f=0.6-0.9,
plausibly real local strain patterns from the interpolation rather than
noise, but not distinguishable from noise at N=1 per point) -- a converged
string method with reparametrization would be needed to smooth this into
the actual minimum free energy path rather than the raw linear-interpolation
path's energy.

### An important tool-4 caveat, found by actually using it here

`landscape_along_path.png` shows tool 4's Boltzmann-weighted KDE landscape
estimator (`gpcr_energy_landscapes.pipeline.build_1d_landscape` with
`gibbs=`) applied to this path -- and it's **wrong**, visibly so (the
scatter of raw per-image energies is plotted on top of it for exactly this
reason). With only 11 points and energy differences (137 kcal/mol) vastly
exceeding RT (~0.6 kcal/mol at 310 K), the Boltzmann weighting collapses
almost entirely onto the single lowest-energy image -- `exp(-137/0.6)` is
astronomically small -- producing a near-flat curve near that one point's
value, not a reflection of the real profile. **This is a genuine
tool-mismatch, not a bug**: `landscape_1d(gibbs=...)` is designed to
combine *many microstates sharing a macrostate* into a population-weighted
free energy (correct for an actual sampled ensemble), not to visualize a
handful of individually-meaningful, already-relaxed path images (where
each point's own G *is* the answer, and `gibbs_vs_path.png`'s direct line
plot is the right way to look at it). Worth remembering when applying tool
4 to small, curated ensembles like this one rather than large sampled ones.

## Known limitations

1. **Not a converged string method** (see above) -- no reparametrization,
   no tangent-force projection. The barrier height/shape should be treated
   as a first estimate of a transition-path profile, not a rigorous
   minimum free energy path or true transition-state energy.
2. **N=1 per interpolation point** -- no error bars, can't distinguish the
   f=0.6-0.9 local wiggle from noise.
3. **Single pH (7.4)** -- the earlier active/inactive-basin run showed a
   large, consistent pH effect; re-running this path at pH 6.0 would show
   whether the barrier height itself is pH-dependent (a natural next step,
   not done here to keep runtime to ~15 min).
4. Same geometric-CV caveats as the other runs (auto-detected, not
   GPCRdb-validated).

```bash
pip install -r requirements.txt
SETUPTOOLS_USE_DISTUTILS=stdlib pip install openmm pdbfixer propka
PYTHONPATH=.:gibbs python examples/gpcr_pipeline_gpr68_string_demo.py
```

Takes ~15-20 minutes (11 real OpenMM minimizations).
